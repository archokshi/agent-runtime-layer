What gets stripped?
The optimizer fingerprints context blocks using SHA-256. Blocks seen in ≥2 consecutive calls are classified as "stable" and flagged for removal from per-call payloads.
Stable: system_prompt (18,400 tok) · tool_defs (8,200 tok)
How is it applied?
POST /api/tasks/{id}/apply-optimization strips the stable blocks, re-runs the optimized payload, and stores an OptimizationProofRecord with before/after delta.
POST /api/tasks/a4f3b1/apply-optimization
Can it be automated?
Yes. Enable "Context Optimizer" in the Control Plane (Settings page). The SDK reads /api/settings on startup and applies auto_optimize=True to every subsequent run automatically.
GET /api/settings → {optimizer_enabled: true}
What does "Verified" mean?
Evidence quality upgrades from "~ Estimated" to "✓ Verified" once a real before/after pair is measured — not just projected from context analysis.

Hooks capture everything

The backend runs entirely locally. All trace data, proof records, budget events, and context memory are stored in a single SQLite file. Zero cloud dependency.

Under the hood
Every component — explained.
🗄 Database schema (SQLite)
trace_events — every hook event captured (event_type, timestamp, attributes, payload)
analysis_reports — per-task analysis (model_time_ms, tool_time_ms, repeated_context_percent, cost)
optimization_proofs — before/after pairs (baseline_tokens, optimized_tokens, evidence_quality)
budget_events — every hook block (cost_at_block, retries_at_block, reason)
context_memory — fingerprinted stable blocks (sha256, token_count, hit_count, cache_savings)
settings — control plane state (plan, optimizer_enabled, budget_enabled, memory_enabled)
🔌 API endpoints (FastAPI :8000)
GET /api/tasks — all traced runs with status, goal, timing
GET /api/tasks/{id}/analysis — bottleneck breakdown, token counts, cost
POST /api/tasks/{id}/apply-optimization — run optimizer, store proof record
GET/PATCH /api/settings — control plane state, plan gates
GET /api/budget/summary — blocked runs, saved dollars
GET /api/context-memory/summary — hit counts, cache savings
🪝 Hook event types captured
PreToolUse — fires before every tool call. Budget Governor checks here and returns blocked=true if limit exceeded.
PostToolUse — fires after tool completes. Captures exit_code, latency_ms, stdout preview.
UserPromptSubmit — fires on every user prompt. Captures session context, token estimate.
Stop — agent run completed or interrupted. Captures final token counts, cost.
🧩 SDK components
tracer.py — AgentRuntimeTracer, reads /api/settings on init, auto_optimize, check_budget()
budget.py — BudgetGovernor, session cost tracking, retry counting
proxy.py — local HTTP proxy :8100, SHA-256 fingerprinting, cache_control injection
integrations/claude_code.py — hook handler, budget enforcement, event capture
integrations/codex.py — same for Codex CLI hooks

Common questions
Everything you want to know.
Does my code or prompt data ever leave my machine?
Never. Agentium runs entirely on localhost. All data is stored in a local SQLite file. The dashboard is a Next.js app served at localhost:4001. No telemetry, no cloud sync, no external APIs called.
Does it change how my agent behaves?
No. The observe mode is completely passive — hooks capture events but don't modify agent behaviour. The optimizer only activates when you explicitly click "Apply Optimization" or enable the toggle in Settings.
What if Docker isn't installed?
The install script detects this and installs Docker automatically. On macOS it uses Homebrew. On Linux it uses apt/yum. On Windows it provides the Docker Desktop download link.
curl -sSL https://agent-runtime-layer.vercel.app/install.sh | bash
How is "−43% token reduction" measured?
The optimizer stores an OptimizationProofRecord with baseline_input_tokens, optimized_input_tokens, evidence_quality ("measured" or "estimated"). The −43% is from a real before/after pair on the same task — not projected.
Can the Budget Governor block my agent mid-run?
Yes. The PreToolUse hook fires before every tool call. If session cost exceeds max_cost_per_run, the hook returns blocked=true and Claude Code stops the run immediately. This is the exact mechanism Claude Code uses for its own safety hooks.
What is Context Memory and how does it save money?
The Context Memory proxy intercepts Anthropic API calls at localhost:8100. It fingerprints stable context blocks (SHA-256), marks them with cache_control: ephemeral, and forwards to Anthropic. Anthropic charges $0.30/MTok for cached tokens vs $3.00/MTok for standard — a 10× discount.

Security & Privacy

Data handling and security model
Your agent data never leaves your machine.
🔒
100% local storage
All trace data, prompts, tool outputs, and optimization results stored in a local SQLite file at ~/.agentium/data/. No external database. No cloud sync. You own everything.
🌐
No external connections
The backend only calls Anthropic API (when proxy is enabled — and only because you set ANTHROPIC_BASE_URL). The dashboard makes no external calls. No telemetry, no analytics, no phone-home.
🔑
Secret redaction
The SDK automatically redacts obvious secrets (API keys, tokens, passwords) before persisting to SQLite. The redaction module runs before any storage — not after. Configurable redaction patterns.
🐳
Docker isolation
All services run in Docker containers. The only network access is localhost:8000 (backend) and localhost:4001 (dashboard). No port exposed externally by default.
📝
Audit trail
Every optimization proof, budget block, and settings change is stored with timestamps. You can see exactly when each optimization was applied and what the before/after delta was.
🗑
Delete anytime
docker compose down removes all services. rm -rf ~/.agentium removes all data. No account to delete, no support ticket required. Complete removal in two commands.
